# Big-O와 입력 크기 — 풀기 전에 복잡도를 "역산"하기

> **한 줄 요약**: 코테에서 Big-O는 "코드를 다 짜고 나서 분석하는 것"이 아니라 **문제의 N 제약을 보고 풀이를 고르기 전에 역산하는 도구**다. "N ≤ 100,000인데 이중 for? → 100억 연산 → 시간 초과" 를 **문제 읽는 단계에서** 판단하는 게 실전 무기.

## 언제 쓰나

- 코테 문제를 읽고 **어떤 접근으로 갈지 정하기 전에** (풀고 나서가 아니다)
- 실무에서 "데이터 늘어나면 이 쿼리/루프 괜찮나?"를 가늠할 때

## 복잡도 감각

빠른 순서. 아래로 갈수록 느리다.

| 표기 | 이름 | 감 | 예시 |
|---|---|---|---|
| O(1) | 상수 | 입력이 커져도 그대로 | HashMap 조회, 배열 인덱스 접근 |
| O(log n) | 로그 | 반씩 줄여나감 | 이분탐색 |
| O(n) | 선형 | 한 번 훑음 | for 한 번 |
| O(n log n) | — | 훑기 × 로그 | 정렬 (`Arrays.sort`) |
| O(n²) | 제곱 | 이중 for | 모든 쌍 비교 |
| O(2ⁿ), O(n!) | 지수/팩토리얼 | 폭발 | 완전탐색, 순열 |

## 핵심 — N으로 역산하는 표

대략 **1초 ≈ 1억 연산** 기준. 문제에 주어진 N 제약을 보고 **허용되는 복잡도를 먼저 정한다.**

| N 크기 | 허용 복잡도 | 신호 |
|---|---|---|
| N ≤ 10~20 | O(2ⁿ), O(n!) | 완전탐색/백트래킹 OK |
| N ≤ 500 | O(n³) | 삼중 for도 됨 |
| N ≤ 5,000 | O(n²) | 이중 for OK |
| N ≤ 100,000 | O(n log n) | **정렬·이분탐색·힙**으로 가야 함 |
| N ≤ 10,000,000 | O(n), O(log n) | 이중 for는 무조건 터짐 |

**읽는 법**: "N이 10만이다 → 이중 for(10만 × 10만 = 100억)는 못 쓴다 → 해시(O(n))나 정렬(O(n log n))로 우회해야 한다."

즉 **N 제약이 곧 힌트다.** 문제 설계자가 "이 복잡도로 풀어라"라고 알려주는 것과 같다.

## ⚠️ 함정

- **자료구조를 "고급"으로 바꿔도 차수는 안 바뀐다.** `String[]`을 `ArrayList`로 바꿔봐야 `contains()`는 여전히 O(n) → 이중 루프면 여전히 O(n²) → 여전히 시간 초과. **차수를 바꾸는 건 자료구조의 "연산 복잡도"이지 "격"이 아니다** (예: 조회를 O(n)→O(1)로 만드는 HashMap).
- **스트림 vs for는 상수 배수 차이일 뿐**, 차수를 바꾸지 못한다. 시간 초과의 원인은 거의 항상 "스트림을 써서"가 아니라 **"O(n²)로 짜서"**다. → 최적화 방향을 잘못 잡으면 시간만 낭비.
- **"큰 입력에서만 실패 = 성능 문제"가 항상 참은 아니다.** 값이 커지면서 동작이 바뀌는 코드(래퍼 캐시 등)일 수도 있다 → [오토박싱 & 래퍼 캐시](../java/basics/autoboxing-wrapper-cache.md)
- 평균과 최악이 다른 경우: HashMap 조회는 **평균** O(1)이지만 해시 충돌이 심하면 최악 O(n) (Java 8+는 트리화로 O(log n)).

## 💡 판단 기준

- **문제를 읽으면 코드보다 N 제약을 먼저 본다.** N을 보고 허용 복잡도를 정하고, 그 복잡도 안에 들어오는 접근만 후보에 올린다. "일단 짜고 안 되면 고친다"는 시험장에서 가장 비싼 전략.
- **막혔을 때 던질 질문은 "더 빠른 문법이 있나?"가 아니라 "차수를 낮출 수 있나?"** — 답은 대개 *"반복 조회를 해시로 바꾼다"*, *"정렬해두고 이분탐색한다"*, *"매번 최소값을 힙에서 꺼낸다"* 셋 중 하나.

### 구체 케이스
프로그래머스 "완주하지 못한 선수": N ≤ 100,000.
- 참가자 명단을 완주자 명단과 **이중 for로 대조** → O(n²) → 100억 연산 → **시간 초과 확정**
- N 제약(10만)을 보고 "O(n) 아니면 O(n log n)이어야 한다"를 **먼저 판단** → 조회를 O(1)로 만드는 **HashMap 카운팅**으로 방향 결정 → O(n) 통과

→ 여기서 핵심은 "HashMap을 안다"가 아니라, **N을 보고 이중 for를 후보에서 먼저 지웠다는 것.**

# 자료구조 선택 — "신호 → 도구" 매핑 (누적 문서)

> **한 줄 요약**: 코테 풀이의 절반은 **"이 상황엔 이 자료구조"라는 신호를 알아채는 것**이다. 풀이를 외우면 며칠 뒤 사라지지만, **신호는 다음 문제로 전이된다.** 문제를 풀 때마다 여기에 **한 줄씩 누적**한다.
>
> ⚠️ 이 문서의 목적은 "풀이 모음"이 아니다. **"어떤 신호를 보고 어떤 도구를 꺼냈는가"**만 남긴다. (코드는 별도 보관 — 이 볼트는 개념 레퍼런스지 솔루션 저장소가 아님)

## 원리 — 신호란 결국 "반복되는 연산"

문제에서 찾을 것은 *"어떤 알고리즘인가"*가 아니라 **"내가 이 문제에서 뭘 반복하게 되나"**다. 반복 연산이 정해지면 자료구조는 **자동으로 따라온다.**

| 반복하는 연산 | 그걸 빠르게 해주는 구조 |
|---|---|
| 계속 **찾는다** | 해시 (O(1) 조회) |
| 계속 **최소/최대를 꺼낸다** | 힙 (O(log n)) |
| 계속 **가장 최근 걸 되돌아본다** | 스택 (O(1)) |
| 계속 **한 겹씩 퍼져나간다** | 큐 / BFS |

## 신호 사전

### 🔍 해시 계열 — "찾기"가 반복될 때

| 문제에 이런 말이 나오면 | 도구 | Java |
|---|---|---|
| "몇 **번** 나왔나", "**개수**", "가장 많이", **동명이인** | **HashMap 카운팅** | `map.merge(k, 1, Integer::sum)` |
| "**중복** 제거", "**종류**가 몇 개", "이미 나왔나" | **HashSet** | `set.add()` · `set.size()` = 종류 수 |
| "짝을 이루는", "A에 대응하는 B" | **HashMap** (key→value) | `map.get(k)` |
| 이중 for로 대조하고 싶어지는데 N이 큼 | **해시로 우회** | O(n²) → O(n) |

> **결정적 신호**: *이중 for를 쓰고 싶은데 N이 크다* → 안쪽 for가 **"찾기"**라면 그건 **해시로 바꾸라는 뜻**이다.

### 📚 스택 / 큐 — "순서"가 의미를 가질 때

| 신호 | 도구 |
|---|---|
| **괄호** 짝 맞추기, 수식 검증 | **Stack** (LIFO) |
| "**되돌리기**", "뒤로가기", "직전 것 취소" | **Stack** |
| "**가장 최근**에 ~한 것" | **Stack** |
| "**최단거리**", "최소 **몇 번** 만에", "몇 칸" | **BFS (Queue)** ⭐ |
| "순서대로 처리", 대기열, 작업 큐 | **Queue** (FIFO) |

> Java는 스택·큐·덱 전부 **`ArrayDeque`** 하나로 해결. (`Stack` 클래스는 레거시 — 안 씀)

### ⛰️ 힙 (우선순위큐) — "매번 최소/최대"

| 신호 | 도구 |
|---|---|
| "가장 작은 것부터 **계속** 꺼내서" | `PriorityQueue` |
| "**K번째** 큰 수", "상위 K개" | `PriorityQueue` |
| "우선순위가 높은 것부터 처리" | `PriorityQueue` |
| 다익스트라 (최단경로 + 가중치) | `PriorityQueue` |

> 기본은 **최소힙**. 최대힙은 `new PriorityQueue<>(Comparator.reverseOrder())`.

### 🧭 그래프 — "연결"

| 신호 | 도구 |
|---|---|
| "**연결**되어 있다", "네트워크", "친구 관계" | 그래프 + DFS/BFS |
| "**그룹이 몇 개**", "섬이 몇 개", "덩어리" | **연결 요소** (DFS/BFS로 세기) |
| 2차원 격자에서 "이동", "퍼진다" | **BFS** (격자도 그래프다) |
| "**최단**" | **BFS** ⭐ |
| "**모든** 경로", "가능한 조합 전부" | **DFS / 백트래킹** |

> 표현: 인접 리스트 `List<List<Integer>>` + 방문 배열 `boolean[]`. 인접 리스트 초기화가 `computeIfAbsent`의 대표 용도 → [Map 주요 메서드](../java/collections/map-methods.md)

### 🔢 정렬 · 이분탐색

| 신호 | 도구 |
|---|---|
| "정렬하면 편해지는" 문제 / 커스텀 순서 | `Arrays.sort` + `Comparator` |
| "**조건을 만족하는 최대/최소값**" + **답의 범위**가 정해짐 | **파라메트릭 서치** (이분탐색) |
| "~이하 중 가장 큰", "~이상 중 가장 작은" | 이분탐색 |

### 📈 DP

| 신호 | 도구 |
|---|---|
| "**경우의 수**가 몇 가지" | DP (점화식) |
| "**이전 결과를 재사용**"할 수 있는 구조 | DP |
| 완전탐색인데 **같은 계산이 반복**됨 | DP (메모이제이션) |

## ⚔️ 자주 헷갈리는 갈림길

**① HashSet vs HashMap**
> **중복이 의미를 갖나?** → 예: **Map**(개수를 센다) / 아니오: **Set**(존재만 본다)
> *"완주하지 못한 선수"는 동명이인이 있어서 Set으론 안 되고 Map이었다.*

**② 정렬 vs 힙**
> **한 번 정렬하고 끝?** → `sort` / **넣고 빼기를 반복?** → `PriorityQueue`
> 중간에 새 값이 계속 들어오는지가 갈림길.

**③ DFS vs BFS**
> **최단거리?** → **BFS** (예외 거의 없음, 외울 것) / **모든 경우·경로 탐색?** → **DFS**

## 🧠 문제 읽을 때 30초 루틴

코드 짜기 **전에** 이 순서로:

1. **N이 얼마?** → 허용 복잡도 역산 (이중 for 되나?) → [Big-O와 입력 크기](./big-o-and-input-size.md)
2. **내가 반복할 연산이 뭐지?** → 찾기 / 최소값 꺼내기 / 최근 것 보기 / 퍼져나가기
3. **그 연산을 빠르게 해주는 구조는?** → 위 사전에서 꺼내기

이 3단계가 붙으면 코테의 절반은 끝난다. 나머지 절반은 구현.

## 구체 케이스 (푼 문제 → 배운 신호)

> 형식: **문제 / 신호 / 도구 / 막힌 지점**

### 완주하지 못한 선수 (프로그래머스 Lv.1) — 2026-07-14
- **신호**: "완주하지 **못한** 선수" = 참가자엔 있는데 완주자엔 없는 사람 → **개수 대조**. + 제약 N ≤ 100,000 → 이중 for(O(n²)) 불가.
- **도구**: `HashMap` 카운팅. 참가자 +1, 완주자 -1 → **0이 아닌 사람이 답**
  ```java
  for (String p : participant) map.merge(p, 1, Integer::sum);
  for (String c : completion)  map.merge(c, -1, Integer::sum);
  // 남은 값이 0이 아닌 key = 정답
  ```
- **막힌 지점 ①**: **HashSet으론 안 된다.** 동명이인 때문에 "있냐 없냐"가 아니라 **"몇 명이냐"**를 세야 함 → *존재 = Set / 개수 = Map* 갈림길.
- **막힌 지점 ②**: `Integer test1 != test2`로 비교 → **127 넘으면 오답**(참조 비교). 정확성은 통과하고 효율성만 실패해서 시간 초과인 줄 알았으나 **실은 오답**. → [오토박싱 & 래퍼 캐시](../java/basics/autoboxing-wrapper-cache.md)
- **막힌 지점 ③**: `computeIfAbsent`로 카운팅하려 함 → **"있을 때" 분기가 없어서 불가.** 카운팅은 `merge`. → [Map 주요 메서드](../java/collections/map-methods.md)

## ⚠️ 함정 (누적)

- **자료구조를 "고급"으로 바꾼다고 빨라지지 않는다.** 배열 → ArrayList는 차수를 안 바꾼다. `ArrayList`가 배열보다 나은 건 **크기가 변할 때뿐.** 바꿔야 할 건 *조회를 O(1)로 만드는 것*(HashMap).
- **HashSet vs HashMap 갈림길**: 중복이 의미를 갖는 문제인지 먼저 확인.
- **스트림 vs for는 상수 차이**일 뿐 차수를 못 바꾼다. 시간 초과의 범인은 거의 항상 O(n²). 최적화 방향을 여기서 잘못 잡으면 시간만 낭비.
- **"최단"인데 DFS로 가면 틀린다.** DFS는 경로를 찾긴 하지만 **최단이라는 보장이 없다.** 최단 = BFS.

## 💡 판단 기준

- **"풀이"가 아니라 "신호"를 외운다.** 풀이는 문제마다 다르지만, *"개수를 물으면 HashMap"*, *"최단거리면 BFS"* 같은 신호는 **모든 문제에 전이된다.** 복습할 때도 코드가 아니라 이 표를 본다.
- **자료구조 선택 기준은 "더 좋은 것"이 아니라 "필요한 연산에 맞는 것".** 내가 반복하는 연산이 *조회*인가, *최소값 꺼내기*인가, *최근 것 되돌아보기*인가 → 그걸 O(1)/O(log n)으로 해주는 구조를 고른다.
- **막혔을 때 던질 질문은 "더 빠른 문법이 있나?"가 아니라 "차수를 낮출 수 있나?"** — 답은 대개 *반복 조회를 해시로*, *정렬해두고 이분탐색으로*, *매번 최소값을 힙에서* 셋 중 하나.
- **복습 = 3일 뒤 백지에서 다시 풀기.** "이해했다"의 증거는 *다시 풀 수 있다*. 노트를 보고 이해되는 건 이해가 아니다.

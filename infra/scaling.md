# 스케일 아웃 & 배포 모델 — 단일 인스턴스에서 분산까지

> **한 줄 요약**: 앱 1개를 실행하면 1개의 프로세스(JVM)로 뜬다. 트래픽이 커지면 같은 앱을 여러 인스턴스로 **복제(스케일 아웃)**하고, 앞단의 **로드밸런서**가 요청을 나눠 보낸다. 이때 "JVM이 여러 개"가 되면서 단일 JVM 가정(공유 메모리·JVM 락·인스턴스 내부 상태)이 깨진다 — 그래서 분산락·무상태 설계가 필요해진다.

관련 노트: [락 개념 종합](../java/concurrency/locks.md) · [JVM 동시성 도구](../java/concurrency/jvm-concurrency-tools.md)

---

## 1. 단일 인스턴스 = 1 JVM

Spring Boot 앱을 실행하면(`java -jar app.jar`) **하나의 프로세스 = 하나의 JVM**으로 뜬다.

```
Spring Boot 인스턴스 1개 (JVM 1개)
 └─ 내장 톰캣(기본)의 스레드 풀 (기본 ~200)
      요청A → 워커 스레드1 ┐
      요청B → 워커 스레드2 ┤ 전부 같은 JVM = 같은 메모리 공유
      요청C → 워커 스레드3 ┘
```

- 들어온 요청은 **스레드 풀의 워커 스레드**가 나눠 처리한다.
- **스프링 빈은 기본 싱글톤** → 인스턴스 하나를 모든 요청 스레드가 공유. 그래서 빈에 가변 상태를 두면 thread-safety 문제가 생기고, JVM 락으로 조율 가능하다.
- 즉 **"한 서버 + 여러 명 동시 호출"은 같은 JVM 안 여러 스레드** → 정상 상황. ("여러 명 호출"이 문제가 아니다)

---

## 2. 스케일 업 vs 스케일 아웃

| | 스케일 업 (Scale Up, 수직) | 스케일 아웃 (Scale Out, 수평) |
|---|---|---|
| 방법 | 한 서버의 사양↑ (CPU/RAM 증설) | 같은 앱을 **여러 인스턴스로 복제** |
| JVM 수 | 그대로 1개 | **여러 개** |
| 한계 | 물리적 상한·비용 급증 | 이론상 무한 확장 (대신 상태 공유 문제) |

> 동시성·락 관점에서 중요한 건 **스케일 아웃** — JVM이 여러 개가 되기 때문.

---

## 3. 스케일 아웃 = 인스턴스 복제 (JVM 여러 개)

트래픽이 커지면 **같은 앱을 N개 인스턴스로 복제**한다. 스프링이 알아서 늘려주지 않는다 — 사람 또는 자동화 도구가 한다.

```
같은 app.jar 를
 서버1: java -jar app.jar   (JVM-1, 메모리 별개)
 서버2: java -jar app.jar   (JVM-2, 메모리 별개)
 (K8s면 pod 2개, 도커면 컨테이너 2개)
```

→ 이 순간부터 "단일 JVM" 가정이 깨진다. 각 인스턴스는 **자기만의 메모리**를 가지므로, 인스턴스 A의 변수·락 객체·캐시를 인스턴스 B는 볼 수 없다.

---

## 4. 로드밸런서 (Load Balancer) — 분배기

여러 인스턴스 앞에 두고 요청을 나눠 보내는 컴포넌트. **스프링 밖의 인프라**다.

```
[사용자들]
   │
[로드밸런서]  ← Nginx / AWS ALB·ELB / K8s Service·Ingress
   ├──→ 서버1 (JVM-1)
   └──→ 서버2 (JVM-2)
```

- **LB는 서버를 만들지 않는다.** 이미 떠 있는 인스턴스 목록(백엔드/타겟)에 요청을 **분배만** 한다. 빈 LB만 있으면 분배할 대상이 없다.
- **순서**: 인스턴스를 먼저 N개 띄움 → LB에 등록 → LB가 분배.
- ⚠️ **LB는 락·상태를 맞춰주지 않는다.** 같은 `userId` 요청이 서버1·서버2로 흩어질 수 있어, 오히려 JVM 락이 깨지는 원인이 된다. (sticky session으로 같은 사용자를 같은 서버에 고정할 순 있지만, 정합성을 그것에 의존하면 안 된다)

> (혼동 주의) **Spring Cloud LoadBalancer**는 *마이크로서비스가 다른 서비스를 호출할 때* 쓰는 클라이언트 측 LB(서비스 디스커버리와 함께)다. 사용자 트래픽 분배나 기본 Spring Boot 기능이 아니다.

---

## 5. 오토스케일러 (Auto Scaler) — 생성기 (LB와 별개)

"자동으로 인스턴스가 늘어나는" 건 LB가 아니라 **오토스케일러**의 일이다.

```
[오토스케일러]  ← K8s HPA / AWS Auto Scaling Group
   CPU·요청량 등 지표 보고 → "부하 높음, 인스턴스 +1" → 서버3 생성
        ↓
   새 인스턴스가 LB 백엔드 목록에 자동 등록
        ↓
[로드밸런서]  → 서버1·2·3 으로 분배
```

| 컴포넌트 | 역할 |
|----------|------|
| **로드밸런서** | 요청 **분배** (서버 생성 X) |
| **오토스케일러** | 부하 보고 인스턴스 **생성/제거** |

→ 둘은 다른 컴포넌트인데 함께 일한다. 오토스케일러가 인스턴스를 만들고/없애면, LB는 그때그때 살아있는 것들로 라우팅한다.

## 6. 스케일 아웃이 코드에 주는 영향 (핵심)

JVM이 여러 개가 되면 "단일 인스턴스 안에서만 통하던 것들"이 깨진다:

| 단일 인스턴스 가정 | 스케일 아웃 시 문제 | 해결 |
|--------------------|---------------------|------|
| 인스턴스 메모리에 상태 저장 (필드·static·로컬 캐시) | 인스턴스마다 값이 다름 | **무상태(stateless) 설계** — 상태는 DB·Redis 등 공유 저장소로 |
| JVM 락(synchronized/ReentrantLock)으로 동시성 제어 | 인스턴스별 락이 따로 놀아 무력 | **분산락**(Redis 등) 또는 **DB 락** → [locks.md](../java/concurrency/locks.md) |
| HttpSession을 메모리에 보관 | 다른 서버로 가면 세션 유실 | 세션 외부화(Redis), sticky session, 또는 토큰(JWT) |
| 로컬 캐시 | 인스턴스 간 캐시 불일치 | 분산 캐시(Redis), 캐시 무효화 전략 |

> 한 줄: **스케일 아웃의 대원칙은 "인스턴스에 상태를 두지 말 것(무상태)".** 상태가 필요하면 모든 인스턴스가 공유하는 외부 저장소로 빼고, 동시성은 그 공유 지점(DB/Redis)에서 제어한다.

---

## 7. 정리

- 앱 1개 = **JVM 1개**. 그 안 여러 스레드로 다중 요청 처리 (Spring 빈은 싱글톤 공유).
- **스케일 아웃 = 인스턴스 복제**(JVM 여러 개). 스프링이 자동으로 안 해준다.
- **로드밸런서 = 분배기**(서버 안 만듦, 외부 인프라). **오토스케일러 = 생성기**(별개 컴포넌트).
- JVM이 여러 개가 되면 인스턴스 내부 상태·JVM 락이 무력 → **무상태 설계 + 분산락/DB 락 + 공유 저장소**.

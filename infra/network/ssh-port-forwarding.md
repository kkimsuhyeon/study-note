# SSH 포트 포워딩 (ssh -L) - 막힌 포트를 "건물 안 손님"으로 우회

> 한 줄 요약: `ssh -L`은 **내 컴퓨터의 빈 포트에 원격 서버로 가는 통로를 뚫는다.** 외부에서 직접 막힌 포트라도, SSH로 일단 서버에 들어간 뒤 "서버 내부에서" 접근하는 효과를 낸다. **함정은 `-L A:B:C`의 가운데 B(localhost)가 "원격 서버 기준"이라는 것.**

## 언제 쓰나

- 원격 서버의 어떤 포트가 **방화벽으로 외부에 막혀 있는데**(→ 직접 접근 timeout), **SSH(22)는 열려 있을 때**
- DB·내부 API·로컬 LLM 등 "localhost에서만 받는" 서비스에 외부에서 붙어야 할 때
- 외부 노출 설정(`0.0.0.0` 바인딩, 방화벽 개방) 없이 안전하게 원격 서비스를 쓰고 싶을 때

## 사용 예시 (문법/명령)

```bash
ssh -L 10531:localhost:10531 test@39.115.161.51
#       └─A─┘ └───B───┘ └─C─┘ └──── 접속할 서버 ────┘
```

문법: `-L [내_컴퓨터_포트]:[대상_호스트]:[대상_포트]`

- **A = 10531** : 내 컴퓨터에 열릴 포트 (여기로 요청을 넣는다)
- **B = localhost** : **원격 서버(39...) 입장에서의** 대상 호스트 ← 함정
- **C = 10531** : 그 대상의 포트 (AI가 실제 listen하는 곳)

실행하면 SSH 세션이 로그인된 채 유지되고(=통로가 열린 상태), 그 동안 내 PC `localhost:10531` → 서버 안 `localhost:10531`로 트래픽이 운반된다.

```bash
# 백그라운드로 통로만 (로그인 셸 없이)
ssh -fN -L 10531:localhost:10531 test@39.115.161.51
#    -f: 백그라운드  -N: 원격 명령 실행 안 함(터널 전용)
```

테스트 (터널 켠 상태, 새 창에서):

```bash
curl http://localhost:10531/v1/models   # 서버의 모델 목록이 나오면 성공
```

## 데이터 흐름

```
[내 PC]                              [원격 서버 39.115.161.51]
ocr/curl → localhost:10531 ─(SSH 암호화 터널)─→ localhost:10531 → AI
           (내 PC의 10531)                       (서버의 10531)
```

내 PC의 10531에 넣었을 뿐인데 실제로는 서버까지 갔다 온다. AI 입장에선 "localhost(서버 자신)에서 온 요청"으로 보여서, 외부 비공개·무인증 경로를 그대로 탄다.

## 함정 / 메커니즘 ⚠️

- **가운데 B는 "원격 서버가 해석하는" 주소다.** `localhost`라고 쓰면 내 PC가 아니라 **접속한 서버 자신**을 가리킨다. 그래서 서버의 AI가 `127.0.0.1`에만 바인딩돼 있어도 터널은 동작한다. (B에 다른 IP를 쓰면, 서버를 경유해 또 다른 호스트로 보내는 것도 가능 = 점프 호스트)
- **터널 = SSH 연결 그 자체.** SSH 창을 닫으면 연결이 끊기고 통로도 사라진다 → 내 10531은 다시 빈 방 → 다시 refused. ocr 돌리는 동안엔 켜둬야 한다(또는 `-fN` 백그라운드).
- **옵션은 반드시 `user@host` 앞에.** `ssh ... host -i key`처럼 호스트 뒤에 쓰면, 뒤의 것이 "원격에서 실행할 명령"으로 해석돼 적용 안 된다. → `ssh -i key -L ... user@host` 순서.
- **`-L`에는 포트(숫자)만.** `-L 10531:localhost:10531/v1/models`처럼 URL 경로를 붙이면 안 된다. 경로는 나중에 curl/ocr이 HTTP 요청 보낼 때 붙이는 것. 터널은 TCP 포트만 잇는다.
- **내 쪽 포트(A)가 이미 점유면** `bind: Address already in use`. 이미 터널이 떠 있거나 다른 프로세스가 10531을 쓰는 경우. (`netstat`로 확인 → 정리)

## 종류 비교 (참고)

| 옵션 | 방향 | 한 줄 |
| --- | --- | --- |
| `-L` (Local) | 내 PC → 원격 | 내 포트로 들어온 걸 원격으로 (이 노트) |
| `-R` (Remote) | 원격 → 내 PC | 원격 포트로 들어온 걸 내 PC로 (반대) |
| `-D` (Dynamic) | SOCKS 프록시 | 동적으로 여러 목적지 (프록시처럼) |

## 💡 판단 기준 (관점)

**"직접 가서 timeout나는데 SSH는 열려 있다"** → 그게 `ssh -L`을 꺼낼 신호다.
- 직접 접근(`curl http://원격IP:포트`)이 **되면** 터널 불필요 — 그냥 직접 쏴라.
- 직접 접근이 **timeout/refused인데 SSH 키는 있다** → 터널. "열쇠 있는 정문으로 들어가 건물 내부에서 접근"하는 게 핵심 비유.
- 외부 노출(`0.0.0.0`+방화벽 개방) vs 터널 중 고민되면 → **터널이 더 안전**. 포트를 인터넷에 까지 않고 SSH 인증에 얹어가니까.

> 실제 케이스: 맥미니 AI(10531)에 내 PC 직접 curl → timeout. 하지만 `ssh mac_mini`는 됨. 그래서 `ssh -L 10531:localhost:10531`로 터널 → 내 PC localhost:10531이 맥미니 AI에 연결됨. ocr의 `llm.url`을 `http://localhost:10531/...`로 두니 동작.

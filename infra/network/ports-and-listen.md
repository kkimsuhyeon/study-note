# 포트와 listen - "연결 거부"와 "응답 없음"이 다른 이유

> 한 줄 요약: 포트는 한 컴퓨터(IP) 안의 "호실 번호"고, 프로그램이 그 방을 잡고 대기하는 게 listen이다. **listen하는 게 없으면 즉시 거부(refused), 방화벽이 막으면 응답 없음(timeout)** — 이 둘은 원인이 완전히 다르다.

## 언제 쓰나 (언제 마주치나)

- `curl`/`telnet`으로 어떤 서버에 붙었는데 실패할 때, **에러 종류로 원인을 좁힐 때**
- 로컬에서 띄운 서버(DB, API, LLM 등)에 접속이 안 될 때 "내가 포트를 잘못 봤나 / 안 떴나 / 막혔나"를 가를 때
- SSH 포트 포워딩·도커 포트 매핑 등 "포트"가 등장하는 모든 상황의 기반 개념

## 사용 예시 (개념/확인 명령)

한 컴퓨터에는 포트가 1~65535번까지 있다. 프로그램이 통신하려면 포트 하나를 잡고 "여기서 듣고 있겠다"고 선언한다 = **listen**.

```
[IP 39.115.161.51 = 건물]
 ├─ 22번 포트   → sshd가 listen (정문)
 ├─ 10531번 포트 → AI 서버가 listen
 └─ 그 외 포트   → 아무도 없음 (빈 방)
```

특정 포트에 누가 듣고 있는지 확인:

```bash
# macOS / Linux
lsof -i :10531
netstat -an | grep 10531

# Windows (PowerShell)
netstat -ano | findstr :10531
```

`LISTENING` 상태로 잡히면 그 방에 프로그램이 앉아있다는 뜻. 아무것도 안 나오면 빈 방.

접속 테스트:

```bash
curl http://localhost:10531/v1/models   # HTTP 서버면
```

## 종류/옵션 비교 — 접속 실패 3종

| 증상 | 원인 | 해석 |
| --- | --- | --- |
| `Connection refused` (즉시) | 그 포트에 listen하는 게 없음 | 빈 방에 노크 → 바로 거절 |
| `Connection timeout` (한참 후) | 방화벽이 패킷을 조용히 버림(drop) | 노크해도 경비원이 무시 |
| `No route to host` | 그 IP 자체에 도달 불가 | 건물 주소를 못 찾음 |

## 함정 / 메커니즘 ⚠️

- **`refused`와 `timeout`은 의미가 정반대다.** refused는 "거기까지는 갔는데 받는 사람이 없다"(서버 미기동·포트 오타), timeout은 "가는 길이 막혔다"(방화벽·외부 비공개). 디버깅 방향이 완전히 달라진다.
- **`127.0.0.1`(localhost)에만 listen하면 외부에서 못 붙는다.** 많은 서버가 기본적으로 자기 자신(loopback)에만 바인딩한다. 외부 공개하려면 `0.0.0.0`(모든 인터페이스)에 바인딩해야 한다. 이게 "로컬에선 되는데 다른 컴퓨터에선 안 되는" 흔한 원인.
- **포트는 IP에 종속이다.** "10531 포트"는 항상 "어느 IP의 10531이냐"가 같이 와야 의미가 있다. 내 컴퓨터의 10531과 원격 서버의 10531은 완전히 다른 방이다. (← 포트 포워딩 이해의 핵심)

## 💡 판단 기준 (관점)

접속이 안 될 때, **에러 메시지부터 보고 방향을 가른다.**
- `refused` → "서버가 떴나? 포트 맞나?" (내가 붙으려는 쪽 문제) → 서버 기동·포트 확인
- `timeout` → "방화벽이나 외부 비공개인가?" (경로 문제) → 직접 접근 대신 **SSH 터널**(→ ssh-port-forwarding.md) 같은 우회 필요

> 실제 케이스: 맥미니의 AI(10531)에 내 PC에서 직접 `curl` → **timeout**. localhost에서 같은 curl → 성공. 즉 서버는 살아있고 외부 직접 접근만 막힌 것. 그래서 SSH 터널로 우회했다. refused였다면 서버를 먼저 의심했을 것.

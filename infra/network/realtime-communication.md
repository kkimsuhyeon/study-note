# 실시간 통신 기법 비교 - Polling / Long Polling / SSE / WebSocket + relay 패턴

> 한 줄 요약: HTTP는 원래 "클라이언트가 물어야 서버가 답하는" 요청-응답 모델이라 서버가 먼저 말을 걸 수 없다. 이를 우회하는 기법이 polling → long polling → SSE → WebSocket 순으로 진화했고, **"양방향이 정말 필요한가"가 SSE vs WebSocket의 갈림길**이다.

## 배경: 왜 이런 기법들이 필요한가

HTTP의 기본 모델은 **요청-응답(request-response)**:

```
클라이언트 ──요청──▶ 서버
클라이언트 ◀──응답── 서버   (여기서 끝. 서버가 먼저 보낼 방법이 없음)
```

- 서버에서 새 데이터가 생겨도(새 채팅, 알림, 주가 변동) **클라이언트가 물어보기 전까지 전달할 방법이 없다.**
- 이 "서버 → 클라이언트 푸시" 문제를 푸는 방법이 아래 4가지.

## 기법 4종 진화 순서

### 1. Polling (폴링)
클라이언트가 주기적으로 계속 물어봄.

```
클라 ──새거 있어?──▶ 서버 ──없어──▶ (연결 종료)
      (3초 후)
클라 ──새거 있어?──▶ 서버 ──없어──▶ (연결 종료)
      (3초 후)
클라 ──새거 있어?──▶ 서버 ──있어! 데이터──▶
```

- 구현 제일 쉬움 (그냥 `setInterval` + GET)
- 단점: 대부분의 요청이 헛방(빈 응답) → 서버/네트워크 낭비, 주기만큼 지연 발생

### 2. Long Polling (롱 폴링)
물어봤을 때 데이터가 없으면 **서버가 응답을 보류하고 기다림**. 데이터가 생기면 그때 응답.

```
클라 ──새거 있어?──▶ 서버 (…데이터 생길 때까지 응답 안 함, 최대 30초 등)
클라 ◀──있어! 데이터── 서버   → 클라는 받자마자 즉시 다시 요청
```

- 폴링보다 지연·헛방 적음. WebSocket 이전 시대의 주력 (예전 카카오톡 웹, Facebook 채팅 초기)
- 단점: 응답 하나 받을 때마다 재연결 오버헤드, 서버는 대기 중 커넥션을 계속 물고 있음

### 3. SSE (Server-Sent Events)
**HTTP 응답을 끝내지 않고** 열어둔 채, 서버가 데이터가 생길 때마다 조금씩 흘려보냄 (단방향 스트림). → 상세: [sse.md](./sse.md)

```
클라 ──구독 요청──▶ 서버
클라 ◀── data: A ── 서버
클라 ◀── data: B ── 서버      (연결 하나로 계속)
클라 ◀── data: C ── 서버
```

### 4. WebSocket
HTTP로 시작해서 **별도 프로토콜로 업그레이드**, 이후 양방향 자유 통신. → 상세: [websocket.md](./websocket.md)

```
클라 ──HTTP Upgrade 요청──▶ 서버
클라 ◀──101 Switching Protocols── 서버
클라 ◀────양방향 메시지 자유롭게────▶ 서버
```

## 비교표

| | Polling | Long Polling | SSE | WebSocket |
|---|---|---|---|---|
| 방향 | 클라 주도 | 클라 주도 | **서버→클라 단방향** | **양방향** |
| 프로토콜 | HTTP | HTTP | HTTP (그대로) | 전용 (ws://, HTTP로 핸드셰이크만) |
| 실시간성 | 낮음 (주기만큼 지연) | 중간 | 높음 | 높음 |
| 서버 부하 | 헛방 요청 많음 | 커넥션 점유 | 커넥션 점유 | 커넥션 점유 |
| 데이터 형식 | 자유 | 자유 | **텍스트(UTF-8)만** | 텍스트 + 바이너리 |
| 자동 재연결 | (매번 새 요청) | 직접 구현 | **브라우저 내장** (EventSource) | 직접 구현 필요 |
| 프록시/LB 통과 | 문제 없음 | 대체로 문제 없음 | 대체로 문제 없음 (버퍼링 주의) | Upgrade 지원 설정 필요 |
| 구현 난이도 | 최하 | 낮음 | 낮음~중간 | 중간~높음 |

## relay(중계) 패턴 — 서버가 소비자이자 생산자

우리 서버가 **외부의 스트림을 받아서(소비) 그대로 클라이언트에게 흘려보내는(생산)** 구조. 프록시처럼 중간에 서서 스트림을 중계한다.

```
외부 API ──SSE──▶ 우리 서버 ──SSE──▶ 프론트
          (스트림 소비자)      (스트림 생산자)
```

**왜 프론트가 외부 API를 직접 안 부르고 중계하나:**
1. **인증/비밀 보호** — 외부 API 토큰·키를 프론트에 노출할 수 없음. 서버가 토큰을 관리하고 헤더를 붙여 대신 호출
2. **부가 처리** — 흘려보내면서 중간에 이벤트를 엿봐서(peek) DB 저장, 로깅, 과금 집계 등을 수행
3. **계약 안정화** — 외부 API 스펙이 바뀌어도 프론트와의 계약은 우리 서버가 방어

**LLM 채팅에서 전형적인 구조**: OpenAI/Anthropic 등 LLM API가 SSE로 토큰을 흘려주면, 백엔드가 이를 구독해 프론트로 pass-through하고, "완료 이벤트"가 오면 전체 답변을 DB에 저장. (relay 없이 프론트가 직접 붙으면 API 키가 노출되고 저장 시점을 잡을 수 없다)

⚠️ relay 서버의 부담: 스트림이 살아있는 동안 **커넥션을 양쪽으로 하나씩 물고 있어야 함** (외부와 1개 + 프론트와 1개). 동시 스트림 수만큼 커넥션이 필요하므로 전용 커넥션 풀 분리, 타임아웃 설계가 중요해진다.

⚠️ relay에서 재시도(retry) 금지: 스트리밍 POST에 자동 retry를 걸면 **비멱등 요청이 중복 실행**될 수 있다 (LLM 질문이 두 번 전송 → 과금 2배, 답변 2개). 스트림 요청엔 retry를 빼고 실패를 그대로 클라이언트에 알리는 편이 안전.

## ⚠️ 공통 함정

- **"실시간 = WebSocket"이 아니다.** 방향을 먼저 따져라. 서버→클라 단방향이면 SSE로 충분하고 인프라도 단순하다.
- 폴링이 항상 나쁜 것도 아니다 — 갱신 주기가 느슨하고(분 단위) 클라이언트가 적으면 폴링이 가장 단순하고 튼튼한 답.
- SSE/WebSocket 모두 **연결을 계속 물고 있는 모델**이라, 스레드-per-요청 서버(전통 서블릿)에서는 비동기 처리(Servlet 3.0 async, reactive)와 조합해야 스레드 고갈을 피한다.

## 💡 판단 기준

- **서버→클라 단방향 푸시(알림, 진행률, LLM 토큰 스트림)** → SSE. HTTP 그대로라 프록시/LB/인증 재사용, 브라우저 자동 재연결까지 공짜.
- **양방향 대화(채팅 입력·게임·협업 편집처럼 클라도 계속 보냄)** → WebSocket.
- **갱신이 느슨(수십 초~분)하고 단순함이 최우선** → 그냥 폴링.
- 구체 케이스: LLM 채팅 "전송" API — 질문은 POST 본문으로 이미 서버에 들어갔고 이후엔 답변이 흘러나오기만 하면 됨 → 양방향 필요 없음 → WebSocket은 과하고 SSE relay가 정답이었다.

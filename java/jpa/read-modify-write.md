# Read-Modify-Write와 트랜잭션 경계 — 쓰기 서비스가 조회 서비스에 의존하면 안 되는 이유

> **한 줄 요약**: "읽은 값을 근거로 변경하는"(read-modify-write) 로직은 **읽기와 쓰기가 같은 트랜잭션 안**에 있어야 한다. 그래서 명령(쓰기) 서비스는 조회(Query) 서비스에 읽기를 위임하지 않고 **자기 트랜잭션 안에서 (필요하면 락 걸어) 직접 읽는다.**

관련 노트: [@Lock 기본](./lock.md) · [@Lock 실무 패턴](./lock-practical.md) · [영속성 컨텍스트](./persistence-context.md) · [락 개념 종합](../concurrency/locks.md)

---

## 1. 핵심 개념 — "쓰기 서비스가 읽기 서비스에 의존하지 않는다"

잔액 충전 같은 메서드를 보자.

```java
@Transactional
public void addBalance(String userId, BigDecimal amount) {
    User user = repository.findByIdForUpdate(userId).orElseThrow();  // ① 락 걸고 읽기
    user.addBalance(amount);                                          // ② 메모리에서 변경
    repository.save(user);                                            // ③ 저장
}
```

①②③이 **한 트랜잭션 안에서 연속으로** 일어난다. 그리고 읽기를 `QueryService`에 맡기지 않고 **자기가 직접** 한다. 이게 "(쓰기 서비스가) 읽기 서비스에 의존하지 않는다"의 의미다.

> 패턴 이름: **read-modify-write** — "읽고(read) → 그 값을 근거로 바꾸고(modify) → 저장(write)". 이 세 단계 사이에 남이 끼어들면 안 된다.

---

## 2. 왜 같은 트랜잭션에서 읽어야 하나 → Lost Update(갱신 손실)

읽기를 **다른(이미 끝난) 트랜잭션**에서 하면 동시성 상황에서 깨진다. 잔액 1000원에 두 요청이 동시에 +500 하는 경우:

```
[읽기를 별도 트랜잭션에서 → 읽고 나서 락이 이미 풀린 상태]
A: 읽음(1000) ─┐
B: 읽음(1000) ─┤  둘 다 1000을 봄
A: 1000+500 = 1500 저장
B: 1000+500 = 1500 저장   ← B는 A의 변경을 못 봄
결과: 1500  (정답은 2000) → 500원 증발 💀
```

```
[같은 트랜잭션 + findByIdForUpdate = SELECT ... FOR UPDATE]
A: 락 잡고 읽음(1000)
B: 락 대기... ⏳
A: 1500 저장 후 커밋 → 락 해제
B: 이제 읽음(1500) → 2000 저장
결과: 2000 ✅
```

> 핵심: **"읽은 값을 근거로 변경하는데(read-modify-write), 읽기와 쓰기 사이에 남이 끼어들면 안 된다."** 락을 잡은 그 트랜잭션이 끝날 때까지 다른 트랜잭션을 막아야 하는데, 읽기를 다른(이미 끝난) 트랜잭션에서 하면 그 보호가 사라진다.

(Lost Update 자체 개념은 [@Lock 기본 §1](./lock.md) 참고)

## 3. 다른 트랜잭션에서 읽으면 생기는 문제 3가지

| # | 문제 | 설명 |
|---|------|------|
| 1 | **Lost Update** | §2 예시. 제일 치명적. 읽고 변경하는 사이 남의 변경이 덮여 사라진다. |
| 2 | **락 조기 해제** | 비관적 락은 **그 트랜잭션이 끝나면 풀린다.** 읽기를 먼저 끝난 트랜잭션에서 하면, 정작 쓸 때쯤엔 락이 이미 풀려서 무방비. |
| 3 | **JPA dirty checking 무력화** | 더티 체킹은 **같은 영속성 컨텍스트(=같은 트랜잭션)** 안에서만 동작. 다른 트랜잭션에서 읽어온 엔티티는 **detached 상태**라 `addBalance()` 해도 자동 반영이 안 된다. |

> 2번이 특히 중요: 락의 생명주기 = 트랜잭션의 생명주기다. "읽을 때 잠깐 락 걸고 트랜잭션 끝, 그 다음 다른 트랜잭션에서 수정"은 보호가 안 된다. **락이 필요한 읽기와 그 읽은 값을 쓰는 행위는 반드시 한 트랜잭션이어야 한다.** (락 생명주기 상세는 [영속성 컨텍스트](./persistence-context.md))

---

## 4. 그럼 "다른 데서 읽어도 되는" 경우는?

- **읽은 값으로 변경 결정을 안 할 때.** 예: 단순 화면 조회, 목록 표시 — 그냥 보여주기만 하면 별도 트랜잭션(`QueryService`)에서 읽어도 OK.
- 즉 **read-modify-write가 아니면** 분리해도 안전하다. 그래서 **조회 전용 서비스(`@Transactional(readOnly = true)`)는 따로 두는** 게 정석.

```
read-modify-write (읽은 값으로 변경)  →  같은 트랜잭션 + 직접 읽기 (필요시 락)
단순 조회 (보여주기만)               →  QueryService로 분리해도 안전
```

## 5. 보너스 — CommandService가 QueryService.getUser()를 부르면?

"읽기 로직 재사용하려고" 명령 서비스가 조회 서비스를 호출하면 어떻게 될까?

Spring 기본 전파(`REQUIRED`)라 **inner 트랜잭션이 outer에 합류**해서 같은 트랜잭션으로 묶이긴 한다. 그런데:

- `getUser`가 `findById`(**락 없음**)라면 → **비관적 락이 안 걸린다** → `addBalance`엔 부적합.
- inner의 `readOnly` 힌트도 **바깥(outer) 트랜잭션이 우선**이라 무시된다.

→ 결국 "재사용하려고 `QueryService` 호출"은 **락도 안 되고 의미도 없다.** 명령 서비스는 자기 트랜잭션 안에서 (필요하면 락 걸어) **직접 로드**하는 게 맞다.

> 정리: 트랜잭션은 합쳐져도 **"어떻게 읽었는가(락 유무)"는 호출한 조회 메서드가 결정**한다. 조회 전용으로 만든 메서드엔 락이 없으니, 그걸 빌려 쓴다고 락이 생기지 않는다.

---

## 6. 정리

- read-modify-write는 **읽기·쓰기가 한 트랜잭션** 안에 있어야 한다 → 그래서 **쓰기 서비스는 읽기를 직접** 한다.
- 다른 트랜잭션에서 읽으면 → ① Lost Update ② 락 조기 해제 ③ dirty checking 무력화(detached).
- **락의 생명주기 = 트랜잭션 생명주기.** 보호가 필요한 읽기와 그 값을 쓰는 행위를 쪼개면 안 된다.
- 단순 조회(보여주기)는 read-modify-write가 아니므로 `QueryService`로 분리해도 안전.
- `QueryService` 재호출은 전파로 트랜잭션은 합쳐지지만 **락이 따라오지 않으므로** 명령 로직엔 부적합.

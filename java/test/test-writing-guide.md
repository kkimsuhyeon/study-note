# 테스트 작성 가이드 — 뭘·어떤 케이스를 검증할지 (성장형 노트)

> **한 줄 요약**: 테스트가 막히는 진짜 이유는 문법이 아니라 **"무엇을 / 어떤 경우를 검증할지 못 떠올리는 것."** 구조(given-when-then) → 검증 대상 4종 → 케이스 도출 체크리스트 순서로 잡으면 막힘이 줄어든다. **내가 막혔던 케이스는 맨 아래에 계속 추가**한다.

관련 노트: [AssertJ 단언](./assertj.md) · [BigDecimal](../bigdecimal/bigdecimal.md) · [@Lock 실무 패턴(동시성 테스트)](../jpa/lock-practical.md)

---

## 1. 막힐 때 따라갈 순서

```
1) 무엇을 테스트? (메서드 하나 = 테스트 여러 개)
2) 어떻게 검증?   → 반환값 / 상태변화 / 상호작용 / 예외 중 무엇?
3) 어떤 케이스?   → 정상(happy) + 경계 + 예외 (아래 체크리스트)
4) given-when-then으로 한 케이스씩 작성
```

> 핵심 마인드셋: **"이 메서드가 약속한 게 뭐지? 그 약속이 깨지는 경우는?"** 테스트는 그 약속을 코드로 박제하는 것.

---

## 2. 구조 — given / when / then (AAA)

```java
@Test
void 잔액을_충전하면_그만큼_늘어난다() {
    // given — 준비 (입력·초기상태)
    User user = new User("u1", BigDecimal.valueOf(1000));

    // when — 실행 (테스트 대상 행위 1개)
    user.addBalance(BigDecimal.valueOf(500));

    // then — 검증
    assertThat(user.getBalance()).isEqualByComparingTo("1500");
}
```

- **when은 한 줄(한 행위)** 이 이상적. 여러 개면 테스트를 쪼개라.
- **then에서 검증이 없으면 테스트가 아니다**(그냥 실행일 뿐). 반드시 `assertThat`.

---

## 3. 무엇을 검증하나 — 4가지 유형

| 유형 | 검증 대상 | 예 |
|------|-----------|-----|
| **반환값** | 메서드가 돌려준 값 | `assertThat(result).isEqualTo(...)` |
| **상태 변화** | 객체 필드가 바뀜 | `assertThat(user.getBalance())...` |
| **예외** | 잘못된 입력에 예외 | `assertThatThrownBy(...)` |
| **상호작용** | 협력 객체를 불렀나(Mock) | `verify(repo).save(...)` |

> 막히면 "이 중 어느 걸 확인해야 하지?"부터 자문. 대부분 **반환값 또는 상태 변화 + 예외**다.

---

## 4. ⭐ 어떤 케이스를 쓰나 — 빠뜨리기 쉬운 것 체크리스트

정상 케이스 하나만 쓰고 끝내는 게 가장 흔한 실수. **경계와 예외가 진짜 버그를 잡는다.**

### 입력값 경계 (특히 숫자/돈)
- [ ] **0** — 0 충전/0 차감 허용? 거부? ([BigDecimal `compareTo(ZERO)`](../bigdecimal/bigdecimal.md))
- [ ] **음수** — 음수 입력 막는가?
- [ ] **경계 바로 위/아래** — 잔액과 *정확히* 같은 금액 차감, 1 모자란 금액
- [ ] **최댓값/오버플로** — 아주 큰 값
- [ ] **null** — null 인자 → NPE? 검증 예외?

### 상태 경계
- [ ] **빈 상태** — 잔액 0에서 차감, 빈 리스트 조회
- [ ] **부족/초과** — 잔액보다 큰 차감 → 예외 나는가? (가장 중요한 도메인 규칙)
- [ ] **없는 대상** — 존재하지 않는 id 조회 → 빈 Optional? 예외?

### 컬렉션/문자열
- [ ] 빈 컬렉션 / 원소 1개 / 여러 개
- [ ] 빈 문자열 `""` / 공백 / 아주 긴 문자열

### 예외 케이스
- [ ] 잘못된 입력마다 **어떤 예외 타입 + 메시지**인지 단언 ([assertThatThrownBy](./assertj.md))
- [ ] 예외가 났을 때 **상태가 안 바뀌었는지**도 확인 (롤백/원자성)

### (해당 시) 동시성
- [ ] 같은 자원에 동시 요청 → 정확히 1개만 성공하나 (→ [동시성 테스트 패턴](../jpa/lock-practical.md))

## 5. 테스트 이름 — "무엇을 하면 어떻게 된다"

```java
// ✅ 행위와 기대결과가 드러남 (한글도 좋음)
void 잔액보다_큰_금액을_차감하면_예외가_발생한다()
void addBalance_negativeAmount_throwsException()

// ❌ 무엇을 검증하는지 모름
void test1()  void addBalanceTest()
```

> 좋은 이름은 **실패했을 때 무엇이 깨졌는지** 바로 알려준다. `메서드_상황_기대결과` 패턴이 무난.

---

## 6. 한 도메인 예 — 잔액(User)로 케이스 펼치기

`deductBalance(amount)` 하나에 테스트가 여러 개 나온다:

```java
정상:   1000원에서 300 차감 → 700
경계0:  0 차감 → 거부(예외) 또는 변화없음 (정책에 따라)
음수:   -100 차감 → IllegalArgumentException
부족:   1000원에서 1500 차감 → 잔액부족 예외
정확:   1000원에서 1000 차감 → 0 (경계 통과)
예외후: 부족 차감 시도 후 잔액이 그대로 1000인지 (상태 불변)
```

> 메서드 1개 → 테스트 5~6개가 정상이다. "정상 1개"로 끝내지 말 것.

## 7. 자주 하는 실수

| 실수 | 고침 |
|------|------|
| 정상 케이스만 작성 | 경계·예외 케이스 추가(§4) |
| `then`에 단언이 없음 | 반드시 `assertThat` |
| BigDecimal을 `isEqualTo`로 비교 | `isEqualByComparingTo` ([why](./assertj.md)) |
| `when`에 여러 행위 | 테스트를 쪼갬 |
| 테스트끼리 상태 공유(순서 의존) | 각 테스트는 독립(given에서 새로 준비) |
| 예외만 보고 상태 검증 누락 | 예외 후 상태 불변도 확인 |

## 8. 📌 내가 막혔던 케이스 (계속 추가)

> 테스트 짜다 막힌 상황을 여기에 기록 — 다음에 같은 데서 안 막히게.

### (예시 틀)
- **상황**: 무엇을 테스트하려다 막혔나
- **막힌 점**: 뭘 몰랐나 / 어떤 케이스를 못 떠올렸나
- **해결**: 어떻게 풀었나 (단언/패턴)

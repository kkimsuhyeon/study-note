# study-note 작성 가이드 (Claude용)

이 저장소는 **학습 노트 볼트(Obsidian)**다. 코드 프로젝트가 아니라 "도구 사용법·개념 레퍼런스"가 본질.

## 노트 작성 원칙

1. **사용법/문법/이론이 본문, 관점은 보조.** API를 "어떻게 쓰는가"가 본문이고, 판단 기준은 ⚠️·💡·표로 덧붙인다. (기준 톤: `java/test/assertj.md`)
2. **끝에 판단 기준(관점) 한 줄을 남긴다.** API 문법은 다시 찾으면 되지만 "어느 쪽을 골라야 하나"는 한 번 잡으면 비슷한 상황 전부에 적용된다 — 제일 오래 사는 층.
3. **관점은 반드시 구체 케이스에 붙여서.** "이 상황에서 이렇게 막혔다 → 그래서 이 판단" 형태로. 관점만 따로 모은 추상 격언집은 죽은 문서가 된다.

### 문서 3층 구조 (아래로 갈수록 오래 산다)
| 층 | 성격 |
|---|---|
| ① API 사용법 | "어떻게 쓰지" — 라이브러리/문법 따라감 |
| ② 함정/메커니즘 | "왜 이렇게 동작하지" — 꽤 오래감 |
| ③ **관점/판단 기준** | "어느 쪽을 골라야 하지" — **제일 오래감 (알맹이)** |

→ API는 ③에 도달하기 위한 예시. 본문(①)으로 시작해 ②⚠️·③💡으로 마무리.

## 캡처 모드 (사용자 지침)
- 사용자는 모든 문서를 즉시 다 읽지 못한다. **공부하다 나온 다양한 개념·함정·판단을 적극적으로 노트에 캡처**한다 — "기록할 가치 있어 보이면 일단 적는다"가 기본.
- 새 주제면 새 노트(+README 링크), 기존 주제면 해당 노트에 섹션/💡 보강. 어디에 무엇을 적었는지 사용자에게 한 줄로 알린다.
- 불필요한 건 **사용자가 나중에 읽으며 정리/삭제**한다. 그러니 "혹시 안 쓸까 봐" 빼지 말고, 애매하면 적어두는 쪽.

## 형식 규칙 (README와 동일)
- 한 개념당 한 파일, 파일명 kebab-case
- 새 글 추가 시 **README에 링크 등록**
- 글 구조: 한 줄 요약 → 언제 쓰나 → 사용 예시(문법) → 옵션 비교 → 함정(⚠️) → 판단 기준(💡) → 참고 + 학습 날짜·계기
- 중복 주의: 이미 다룬 메커니즘은 새로 쓰지 말고 **해당 노트로 링크** (예: flush 메커니즘 → `java/jpa/persistence-context.md`)

## 파일 작업
- 노트는 네이티브 Write/Edit로 작성 (이 저장소는 한국어로 서술)
